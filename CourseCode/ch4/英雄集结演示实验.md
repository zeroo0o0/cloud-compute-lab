# 英雄集结演示实验

### 实验一：突破单线程瓶颈（Goroutine 并发改造）

- **对应页码：** 第 4-6, 11-12 页，15页
- **测试目标：** 验证单线程阻塞的“卡顿传染”现象 ，以及 Goroutine 如何通过极低的内存开销（初始 2KB 栈空间 ）实现并发解耦。
- **操作步骤：**
  1. 运行单线程版本，模拟 4 名玩家，并在处理“玩家4”的输入时强制注入 `500ms` 的延迟（模拟地铁环境断流） 。
  2. 观察这 1 帧的整体耗时 。
  3. 将处理连接的代码修改为多协程并发版本：`go s.handleConn(conn)` 。
- **成功标准：**
  - **失败预期：** 单线程下，一人卡顿全员排队，该帧总耗时超过 523ms 。
  - **成功标准：** 改为 Goroutine 后，流畅玩家的请求被独立处理不再被拖累 ，主循环耗时恢复正常（目标 <100ms ）。

- **注意事项**：改进版本尤其要注意事件驱动+增量同步（P15）的实现，如果觉得跨度过大，可以准备三个版本，中间预留出一个迭代的版本。

### 实验二：临界区与数据竞争（NPC 抢宝复现与修复）

- **对应页码：** 第 17-21 页
- **测试目标：** 复现并发环境下的竞态条件（Race Condition）导致的数据错误 ，并用互斥锁（Mutex）修复。
- **操作步骤：**
  1. 编写无锁版的 NPC 触发逻辑（检查 `if npc.Active` -> 设置 `npc.Active = true`） 。
  2. 利用多个 Goroutine 模拟多名玩家同时抢夺同一个 NPC 宝物 。
  3. 引入 `sync.Mutex`，在检查和修改代码前后加上 `Lock()` 和 `Unlock()` 。
- **成功标准：** 无锁状态下产生“物品被复制”的违反唯一性约束错误 ；加锁后严格保证同一时刻只有一个玩家成功触发，其余玩家放弃 。

### 实验三：告别忙等（条件变量 Cond 与虚假唤醒）

- **对应页码：** 第 23-26, 42 页
- **测试目标：** 解决单纯互斥锁导致的“忙等”消耗 CPU 的问题 ，验证条件变量（Condition Variable）的睡眠等待与通知唤醒机制 。
- **操作步骤：**
  1. **错误演示：** 玩家在宝物槽为空时，利用 Mutex 疯狂执行 Lock -> Check -> Unlock 的循环检查（忙等） 。
  2. **正确改造：** 绑定 `sync.Cond`，宝物空时调用 `npc.cond.Wait()` 释放锁并进入睡觉状态 。
  3. **唤醒：** 模拟另一个协程（NPC 补货），在放入宝物后调用 `npc.cond.Signal()` 随机叫醒一个玩家 。
- **成功标准：** 替换为 Cond 后，由于阻塞休眠，CPU 占用率骤降；同时，代码中必须使用 `for` 循环（如 `for npc.Items == 0`）重新检查条件，成功防止虚假唤醒（Spurious Wakeup）导致拿空队列而报错的情况 。

### 实验四：锁的进阶技巧与粒度优化

- **对应页码：** 第 45-46, 49-51 页
- **测试目标：** 体验 Go 特有的高级并发技巧，验证减小锁粒度（分读写、分对象）对吞吐量的巨大提升。
- **操作步骤：**
  1. **信号量：** 使用容量为 N 的 Channel（如 `make(chan struct{}, 10)`）实现最大并发连接限制 。
  2. **超时锁：** 结合 `select` 和 `time.After` 实现带超时的非阻塞尝试加锁 。
  3. **粒度优化：** 对比“全局大锁”与读多写少场景下的 `sync.RWMutex` 。
- **成功标准：** 获取锁超时的玩家能正确执行降级逻辑（`doFallback()`）不被永远卡死 ；使用读写锁后，多个 `RLock()` 读者可以同时进入临界区执行读取 。

### 实验五：高并发性能榨取（对象池 sync.Pool）

- **对应页码：** 第 54, 56, 58 页
- **测试目标：** 降低高并发下的 GC（垃圾回收）压力，复用临时对象以消除 STW（Stop The World）延迟抖动 。
- **操作步骤：**
  1. 在高频请求场景中，每次都 `new(bytes.Buffer)` 创建新的 10KB 临时缓冲。
  2. 引入 `sync.Pool` 维护对象缓存，通过 `bufferPool.Get()` 获取并重置缓冲，处理完后 `Put()` 放回。
  3. 对比两版的总耗时、内存分配次数、GC 次数和尾延迟。
- **成功标准：** 引入 `sync.Pool` 后，分配次数与 GC 次数显著下降，P99 或 P99.9 延迟明显优于基线版；本实验只讲对象池，数据库连接池保留为扩展概念。

### 实验六：游戏数据分层存储架构（Redis + PostgreSQL）

- **对应页码：** 第 61-63 页
- **测试目标：** 针对高频热数据（如实时排行）和低频冷数据（如配置、历史成就） ，实现正确的数据一致性同步策略。
- **操作步骤：**
  1. **强一致场景（写穿 Write Through）：** 编写金币扣除逻辑，必须先写 DB 成功，再同步更新 Redis，两者皆成功才返回 。
  2. **最终一致场景（旁路缓存 Cache Aside）：** 编写游戏配置表读取逻辑，先查缓存，miss 则查 DB 并回填 。写操作先写 DB，**然后直接删除缓存** 。
- **成功标准：** 成功实现分层架构，能够稳定观察到 Write Through 的双写一致性，以及 Cache Aside 的“首次 miss、回填、再次命中”流程。
