# 英雄集结演示实验

### 实验一：突破单线程瓶颈（Goroutine 并发改造）

- **对应页码：** 第 4-6, 11-12 页，15页
- **测试目标：** 验证服务器串行收包导致的“卡顿传染”现象，以及 Goroutine 如何通过独立执行流实现收包解耦；进一步观察事件驱动主循环如何做到“有事件就处理，没事件不等待”，并只同步变化过的状态。
- **操作步骤：**
  1. 先运行 `01_basic_serial_blocking_demo`，只看 `slowInput` 与 `fastInput` 两个输入源，观察“主循环先等 slow，所以 fast 也被拖慢”的因果关系。
  2. 再运行 `02_network_serial_warzone/server` + `02_network_serial_warzone/client`，把同样的串行读取嵌入 Warzone 的“玩家 ACTION -> 权威 PlayerState -> STATE_UPDATE”极简流程，观察慢玩家如何拖住快玩家。
  3. 运行 `03_basic_goroutine_receiver_demo`，只看 `go receive(...)` 如何把等待从主循环拆到独立 goroutine。
  4. 再运行 `04_network_goroutine_warzone/server` + `04_network_goroutine_warzone/client`，观察 slow 仍然晚到，但 fast 的 ACTION 可以先被应用到权威世界状态。
  5. 运行 `05_basic_event_driven_demo`，只看 tick 中的 `select/default`：有事件就处理，没有事件就继续下一 tick。
  6. 再运行 `06_network_event_driven_warzone/server` + `06_network_event_driven_warzone/client`，观察事件驱动和 `dirtyPlayers` 增量同步如何解决“慢玩家卡住整帧”和“每帧全量广播”的问题。
- **成功标准：**
  - **失败预期：** 串行收包下，fast 输入虽然先到，但因为服务端先读 slow，fast 仍然要排队到 slow 返回之后。
  - **成功标准：** 改为 Goroutine 后，slow 只阻塞自己的收包 goroutine；改为事件驱动后，没有事件的 tick 不等待，只有发生变化的玩家会进入增量同步。

- **注意事项**：实验一的 server 和 client 已拆成独立入口，讲服务端知识点时只看 `*_server/main.go`，讲客户端如何制造快慢输入时只看 `*_client/main.go`。更细的“知识点对应哪块代码”见 `cmd/exp1/README.md`；网络版可以用 `-delay-ms` 调整慢玩家延迟，便于课堂现场投屏。

### 实验二：临界区与数据竞争（NPC 抢宝复现与修复）

- **对应页码：** 第 17-21 页
- **测试目标：** 复现并发环境下的竞态条件（Race Condition）导致的数据错误 ，并用互斥锁（Mutex）修复。
- **操作步骤：**
  1. 编写无锁版的 NPC 触发逻辑（检查 `if npc.Active` -> 设置 `npc.Active = true`） 。
  2. 利用多个 Goroutine 模拟多名玩家同时抢夺同一个 NPC 宝物 。
  3. 引入 `sync.Mutex`，在检查和修改代码前后加上 `Lock()` 和 `Unlock()` 。
- **成功标准：** 无锁状态下产生“物品被复制”的违反唯一性约束错误 ；加锁后严格保证同一时刻只有一个玩家成功触发，其余玩家放弃 。

### 实验三：告别忙等（条件变量 Cond 与虚假唤醒）

- **对应页码：** 第 23-26, 42 页
- **测试目标：** 解决单纯互斥锁导致的“忙等”消耗 CPU 的问题 ，验证条件变量（Condition Variable）的睡眠等待与通知唤醒机制 。
- **操作步骤：**
  1. **错误演示：** 玩家在宝物槽为空时，利用 Mutex 疯狂执行 Lock -> Check -> Unlock 的循环检查（忙等） 。
  2. **正确改造：** 绑定 `sync.Cond`，宝物空时调用 `npc.cond.Wait()` 释放锁并进入睡觉状态 。
  3. **唤醒：** 将“NPC 补货”改成交给演示者操作，在放入宝物后调用 `npc.cond.Signal()` 随机叫醒一个玩家 。
  4. **虚假唤醒极简演示：** 运行 `spurious_wakeup_demo`，对比 `if` 版本被一次“没有补货的通知”唤醒后直接拿空队列，以及 `for` 版本被唤醒后重新检查条件并继续等待。
- **成功标准：** 替换为 Cond 后，由于阻塞休眠，CPU 占用率骤降；同时，代码中必须使用 `for` 循环（如 `for npc.Items == 0`）重新检查条件，成功防止虚假唤醒（Spurious Wakeup）导致拿空队列而报错的情况 。

### 实验四：锁的进阶技巧与粒度优化

- **对应页码：** 第 45-46, 49-51 页
- **测试目标：** 体验 Go 特有的高级并发技巧，验证减小锁粒度（分读写、分对象）对吞吐量的巨大提升。
- **操作步骤：**
  1. **锁代价：** 运行 `lock_cost_demo`，对比“每次击杀都加锁更新共享金币”和“本地累计后再合并一次”的耗时差异，直观看到锁加多了会拖慢程序。
  2. **信号量：** 使用容量为 N 的 Channel（如 `make(chan struct{}, 10)`）实现最大并发连接限制 。
  3. **超时锁：** 结合 `select` 和 `time.After` 实现带超时的非阻塞尝试加锁 。
  4. **粒度优化：** 先演示“没有读写锁”的效果，暂停后再对比读多写少场景下的 `sync.RWMutex` 。
- **成功标准：** `lock_cost_demo` 中频繁加锁版本明显慢于减少加锁次数版本；获取锁超时的玩家能正确执行降级逻辑（`doFallback()`）不被永远卡死 ；使用读写锁后，多个 `RLock()` 读者可以同时进入临界区执行读取 。

### 实验五：高并发性能榨取（对象池、连接池与性能观测）

- **对应页码：** 第 54, 56, 58 页
- **测试目标：** 降低高并发下的 GC（垃圾回收）压力，复用临时对象以消除 STW（Stop The World）延迟抖动；同时用本机 TCP 演示短连接改成长连接池后，如何节省重复建连和断连时间；最后通过 pprof、Benchmark 等工具定位 CPU 热点、内存分配、锁竞争和 goroutine 泄漏 。
- **操作步骤：**
  1. 在高频请求场景中，每次都 `new(bytes.Buffer)` 创建新的 10KB 临时缓冲。
  2. 引入 `sync.Pool` 维护对象缓存，通过 `bufferPool.Get()` 获取并重置缓冲，处理完后 `Put()` 放回。
  3. 将“优化前”和“优化后”分开运行，对比两版的总耗时、内存分配次数、GC 次数和尾延迟。
  4. 运行 `connection_pool_demo`，对比每个请求都新建 TCP 短连接与按 `maxOpenConns / maxIdleConns / connMaxLifetime` 配置复用长连接的建连次数和总耗时。
  5. 运行 `perf_observe_demo` 目录下的教程命令，分别生成 CPU、Heap、Mutex 和 Goroutine 观测结果，并对照 Bad/Good 版本理解如何修复。
- **配套说明：** `CourseCode/ch4/cmd/exp5/perf_observe_demo/README.md` 提供了完整的环境准备、工具命令、观测指标与演示顺序，可作为该部分实验的详细操作手册。
- **成功标准：** 引入 `sync.Pool` 后，分配次数与 GC 次数显著下降，P99 或 P99.9 延迟明显优于基线版；连接池演示中，长连接池版本的建连次数明显少于短连接版本，总耗时也更低；性能观测教程中能够用 `go tool pprof -top` 或 `-benchmem` 指出热点函数、分配来源、锁阻塞位置和 goroutine 泄漏趋势。

### 实验六：游戏数据分层存储架构（先简化版，后 Redis + PostgreSQL）

- **对应页码：** 第 61-63 页
- **测试目标：** 针对高频热数据（如实时排行）和低频冷数据（如配置、历史成就） ，实现正确的数据一致性同步策略。
- **建议讲法：** 先运行 `write_through_simple_demo` / `cache_aside_simple_demo`，只讲“持久层 + 缓存层”的分层思路和访问顺序；学生接受这个心智模型后，再切到真实 Redis + PostgreSQL 完整版。
- **操作步骤：**
  1. **强一致场景（写穿 Write Through）：** 编写金币扣除逻辑，必须先写 DB 成功，再同步更新 Redis，两者皆成功才返回 。
  2. **最终一致场景（旁路缓存 Cache Aside）：** 编写游戏配置表读取逻辑，先查缓存，miss 则查 DB 并回填 。写操作先写 DB，**然后直接删除缓存** 。
- **演示建议：** 将 `Write Through` 和 `Cache Aside` 拆成两个独立程序分别运行，避免课堂输出混在一起。简化版适合这一节先讲架构分层；如果要进一步证明真实中间件中的数据变化，再切完整版，并在首次运行前执行 `go mod tidy`、确认 Docker、Redis、PostgreSQL 已启动。数据库相关实验最好再补充 docker 查库命令，让演示者能直接看到程序运行前后 PostgreSQL / Redis 里的变化。
- **成功标准：** 成功实现分层架构，能够稳定观察到 Write Through 的双写一致性，以及 Cache Aside 的“首次 miss、回填、再次命中”流程。
