# 裂土封疆演示实验

### 实验一：微服务“三分”架构改造 (网络调用替代本地调用)

- **对应页码：** 第 14-17 页 
- **测试目标：** 验证将单体游戏进程拆分为“网关 (Gateway)”、“游戏逻辑 (Game)”和“存储 (Storage)”三个独立进程后的协作能力 。
- **操作步骤：**
  1. 开启三个终端，分别在 8080、8081、8082 端口启动网关、游戏逻辑和存储服务 。
  2. 客户端向网关 8080 发送 `move` 移动指令 。
  3. **制造故障：** 强行关掉游戏逻辑进程 (8081)，然后重启它。
- **观察到什么算成功：**
  - **正常流转成功：** 客户端请求 8080，8080 成功将 HTTP 请求转发给 8081，8081 成功通过 GET/POST 读写 8082 里的玩家坐标 。
  - **解耦成功：** 游戏逻辑进程 (8081) 重启后，玩家的坐标数据依然安全地保存在 8082 中，网关 (8080) 也不需要重启，完美展示了业务与存储解耦、部署独立的优势 。

### 实验二:  分片原型 (路由与横向扩展)

- **对应页码：** 第 27-30 页
- **测试目标：** 验证在微服务基础上引入横向分片（Sharding），通过路由机制将不同玩家/地图的流量分发到对应实例，突破单机性能瓶颈。
- **操作步骤：**
  1. 启动 1 个 Web 接入层网关，2 个 Game 计算分片（`Game-0`, `Game-1`），2 个 Storage 存储分片（`Storage-0`, `Storage-1`）。
  2. 构造包含不同 `MapID` 和 `UserID` 的客户端请求，发送给网关。
- **观察到什么算成功：**
  - **精准路由：** 观察日志确认，`MapID=0` 的请求被网关准确路由到了 `Game-0`，随后 `Game-0` 处理逻辑时，对于 `UserID=1` 的数据读取准确请求了 `Storage-1`（假设基于 `%2` 取模）。证明多实例并行工作互不干扰，具备了水平扩展的基础。

### 实验三：一致性哈希 (平滑扩缩容与数据迁移)

- **对应页码：** 第 39-41 页 
- **测试目标：** 解决普通哈希取模（如 `% 4` 变 `% 5`）导致全局数据大洗牌的灾难，验证一致性哈希在扩容时的局部性 。
- **操作步骤：**
  1. 运行代码 `BuildRing`，初始化 4 个物理节点，并为每个节点创建多个虚拟节点（如 `replicas = 10`）打散分布在环上 。
  2. 生成 10 万(适当数量)个测试玩家 ID，通过 `Lookup` 路由，打印出每个节点的玩家分布数量（验证负载是否均匀） 。
  3. 调用 `MigrateRanges` 扩容加入第 5 个节点 。
- **观察到什么算成功：**
  - 加入新节点后，只有与新节点在哈希环上相邻的极少部分（约 1/5）玩家数据被标记需要迁移 。绝大部分旧玩家依然映射在原来的老节点上，证明一致性哈希成功控制了“爆炸半径”。

### 实验四：三种迁移模式的对比 (无感知热迁移)

- **对应页码：** 第 42-46 页
- **测试目标：** 对比不同对象迁移策略在面对海量状态数据时的“停机时间（Downtime）”，验证分波迁移对玩家体验的提升。
- **操作步骤：**
  1. 在代码中模拟一个占用 1000MB 内存状态的“重度玩家”需要跨服迁移。
  2. 运行 **停止复制法（Stop-and-Copy）**：停机 -> 序列化全部 1000MB 传输 -> 恢复。
  3. 运行 **预复制法（Pre-Copy）**：边玩边传 1000MB -> 停机 -> 传输最后 80MB 脏页 -> 恢复。
  4. 运行 **优先级分波迁移（Wave-Based）**：预热静态资源 -> 停机瞬间仅传位置/血量等极少核心状态（10MB） -> 切连接恢复玩家操作 -> 后台静默同步剩余视野外对象。
- **观察到什么算成功：**
  - 控制台日志对比显示：停止复制法导致该玩家卡顿数秒；预复制法卡顿大幅缩小至百毫秒级；而分波迁移法的阻断时间 `< 25ms`，在单帧内完成了关键状态交接，其余数据在后台静默补全。成功验证了极低延迟的无感迁移机制。

### 实验五：Gossip 协议 (世界 Boss 状态广播)

- **对应页码：** 第 59 页 
- **测试目标：** 演示去中心化的病毒式传播，如何在没有中心节点的情况下，让全网数百个服务器迅速同步“世界 Boss 刷新”的消息 。
- **操作步骤：**
  1. 单机启动 100 个轻量级 Goroutine 模拟 100 台服务器。
  2. 向其中 1 台服务器注入 `HasBoss = true` 的状态 。
  3. 观察控制台：每个知道 Boss 存在的服务器，每秒随机挑选 2 个（`fanout=2`）邻居进行 `gossipTo` 异步通知 。
- **观察到什么算成功：**
  - 控制台日志会呈现指数级爆发。大约只需 $O(\log N)$（即 6-7 轮）的极短时间内，100 台服务器全部打印出“已发现世界 Boss”，没有一台服务器因为带宽被打满而崩溃，体现出极高的传染效率 。

### 实验六：分布式事务 Two-Phase Commit (跨服交易极端容错)

- **对应页码：** 第 63-77 页

- **测试目标：** 验证 2PC 协议的“同生共死”原子性，以及严格的状态机流转在应对各类网络与节点崩溃时的兜底能力。

- **推荐演示模式：** 使用剧情演绎式输出（电影化旁白 + 角色台词）展示 2PC 关键转折，并配合状态结果观察 `Coordinator / Worker` 的一致性收敛。

- **注意事项：** 代码必须严格实现包含 `INIT`、`WAIT`、`READY`、`COMMIT`、`ABORT` 的有限状态机架构。

  - 每个状态的转换前后，必须配有模拟持久化日志（`Stable Storage`）写入的 Print 打印。

  *该实验复杂度有些高,如果做demo过于复杂,可以与我进一步交流*

- **操作步骤与故障演示：**

  1. **正常演示 (P67)：** 触发 `START_TX`，Coordinator 与 2 个 Worker 正常完成一二阶段，状态机闭环至 `COMMIT`。
  2. **故障 A (Worker 拒票 P68)：** 故意让 Worker B 在 `INIT` 态判断自身没钱，回复 `VOTE-ABORT`。观察 Coordinator 立即发送 `GLOBAL-ABORT`，Worker A 回滚。
  3. **故障 B (Worker 宕机/超时 P70)：** 拦截 Worker B 的网络响应。观察 Coordinator 在 `WAIT` 状态由于触发 timeout 超时，主动广播 `GLOBAL-ABORT` 防止死锁。
  4. **故障 C (Coordinator 第一阶段宕机 P72)：** 让 Coordinator 在发送 `VOTE-REQ` 途中崩溃。观察 Worker 端在 `INIT` 状态超时后，自动流转至 `ABORT` 并结束。
  5. **故障 D (Coordinator 第二阶段宕机 P73-绝境模拟)：** 最致命的场景！让 Coordinator 收齐所有的 `VOTE-COMMIT`，但在发送 `GLOBAL-COMMIT` 的瞬间强制 Kill 掉它。

- **观察与分析标准 (针对故障 D)：**

  - **现象：** 观察到所有 Worker 进入了 `READY` 状态并**陷入死锁阻塞（Block）**，它们既不敢提交也不敢回滚。
  - **分析与恢复：** 在实验文档中指导学生查看 Coordinator 崩溃前留下的硬盘日志。随后重启 Coordinator 进程，观察其如何执行 `loadPersistedState()` 从日志中发现自身在崩溃前已决定提交，并**重新发送** `GLOBAL-COMMIT` 唤醒所有卡在 `READY` 态的 Worker，最终完成迟到的事务，验证日志+重试是实现“最终一致性”的基石。

### 实验七：Raft 领导者选举 (高可用防宕机)

- **对应页码：** 第 99 页
- **测试目标：** 模拟三个节点组成的 Raft 迷你集群，验证分布式共识算法的自动选举和 Leader 崩溃后的快速故障转移能力。
- **操作步骤：**
  1. 启动 3 个节点，初始状态全为 `Follower`，并各自启动随机时间的 `electionTimeout` 计时器。
  2. 观察谁的随机倒计时先结束，谁转为 `Candidate`，发起 `VoteRequest` 并当选为 `Leader`，随后开始发送心跳（Heartbeat）。
  3. **模拟宕机：** 直接 `Kill` 掉当前的 Leader 进程。
- **观察到什么算成功：**
  - Leader 被杀掉后，剩下的两个 Follower 将收不到心跳信号。它们的计时器恢复走动并很快超时。控制台会迅速打印出幸存者发起新一轮选举，并声明“当选新 Leader，任期 Term = 2”。整个系统在无人工干预的情况下，几百毫秒内自动恢复了合法领袖和可用状态。
