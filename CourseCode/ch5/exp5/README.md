# 实验五：Gossip 协议

本实验用单机 Goroutine 模拟 100 台游戏服务器，演示“世界 Boss 刷新”消息如何在没有中心节点的情况下，通过 Gossip 协议快速扩散到全网。

实验重点是观察 Gossip 的三个核心特征：

- 去中心化：没有一台服务器负责统一广播。
- 病毒式传播：知道消息的服务器会继续通知其他服务器。
- 限制带宽：每台服务器每轮只随机通知 `fanout=2` 个邻居。

## 运行方式

进入实验目录：

```powershell
cd exp5
```

启动演示：

```powershell
go run .
```

程序会自动完成以下步骤：

1. 启动 100 个轻量级 Goroutine，模拟 100 台服务器。
2. 向 `Server-000` 注入 `HasBoss = true`。
3. 每台已发现 Boss 的服务器，每轮随机挑选 2 个不同邻居，通过 `gossipTo` 异步通知。
4. 控制台持续打印每台服务器首次发现 Boss 的日志，以及每轮全网同步进度。

## 关键参数

这些参数都在 `main.go` 顶部，课堂演示时可以直接修改：

```go
const (
    serverCount = 100
    fanout      = 2
    rounds      = 12
)
```

- `serverCount`：模拟服务器数量，默认 100。
- `fanout`：每台已知服务器每轮通知的邻居数量，默认 2。
- `rounds`：最多观察多少轮。
- `tickInterval`：每轮间隔。默认1s。

## 观察什么算成功

运行后会看到类似输出：

```text
启动 100 台服务器，fanout=2，每轮间隔=1s
向 Server-000 注入 HasBoss=true，观察消息如何去中心化扩散。
[第 00 轮] Server-000 已发现世界 Boss：本机注入 HasBoss=true
[第 01 轮] Server-037 已发现世界 Boss：来自 Server-000 的 gossip
[第 01 轮汇总] 已发现世界 Boss 的服务器：  3/100
[第 02 轮汇总] 已发现世界 Boss 的服务器：  8/100
...
全部服务器已同步 Boss 状态，总轮数=8，接近 O(log N) 的传播效果。
```

成功现象：

- 日志不是线性增长，而是前几轮越来越快，呈现指数级扩散趋势。
- 大约 6 到 8 轮左右，100 台服务器全部打印出“已发现世界 Boss”。实际轮数会受随机挑选结果影响。
- 每台服务器每轮只发 2 次异步通知，没有中心节点，也不会出现单点广播导致的带宽打满。

## 实验过程说明

本实验可以把 100 台服务器想象成 100 个互相独立的游戏服：

```text
第 0 轮：只有 Server-000 知道 Boss 刷新
第 1 轮：Server-000 随机通知 2 台服务器
第 2 轮：已经知道 Boss 的服务器继续各自随机通知 2 台
第 3 轮以后：知道消息的服务器越来越多，扩散速度明显加快
```

程序中没有中心广播节点，也没有“总控服务器”负责把消息推给所有人。主函数只负责启动服务器、注入初始状态、打印每轮统计；真正的传播发生在每台服务器自己的 goroutine 里。

课堂演示时建议重点观察两类日志：

- `Server-xxx 已发现世界 Boss`：表示某台服务器第一次收到消息。
- `第 xx 轮汇总`：表示当前全网已有多少台服务器同步成功。

最后的“首次发现轮次分布”可以用来说明传播速度不是线性增加的。前几轮数量可能不大，但一旦更多服务器知道 Boss，它们会同时继续 gossip，传播曲线就会快速抬升。
