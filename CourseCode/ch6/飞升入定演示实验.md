# 飞升入定演示实验

*（后几个实验都和k8s相关且ppt本身已经和教学几乎无异，可以按照ppt做步骤，不严格遵照该文档）*

## 实验一：容器化与环境隔离

- 目标：验证宿主机环境缺失时，容器仍可携带完整运行环境。
- 目录：`exp1`
- 操作：
  1. 在宿主机运行 `go run ./cmd/env_demo`，观察缺少 `/etc/game/config.yaml`。
  2. 构建 `game-map0:v1.0`。
  3. 运行 `docker run --rm game-map0:v1.0`。
- 成功标准：
  - 宿主机运行失败
  - 容器打印 `Server Started`

## 实验二：数据卷持久化

- 对应页码：第 20 页
- 目标：验证容器删除后，宿主机目录中的日志仍保留。
- 目录：`exp2`
- 操作：
  1. 将宿主机目录挂载到 `/app/data`。
  2. 主动访问 `/login` 生成 `players.log`。
  3. 删除容器。
  4. 在宿主机再次查看 `players.log`。
- 成功标准：
  - `players.log` 已生成
  - 删除容器后文件仍存在

## 实验三：进入容器排查问题

- 对应页码：第 21 页
- 目标：验证可使用 `docker exec` 查看容器内进程、端口、网络和日志。
- 目录：`exp3`
- 操作：
  1. 独立启动 `storage` 与 `game-service`，不依赖实验四。
  2. 发送一次 `move` 请求生成日志。
  3. 进入 `game-service` 容器。
  4. 执行 `ps aux`。
  5. 执行 `netstat -tlnp`。
  6. 执行 `ping storage`。
  7. 执行 `cat /app/data/players.log`。
- 成功标准：
  - 可见 `game` 进程
  - 可见 `8081` 监听
  - `storage` 可解析并连通
  - 日志文件可读取

## 实验四：Docker Compose 单机多容器编排

- 对应页码：第 27-29 页
- 目标：使用 `docker-compose.yml` 一次性启动 `storage`、`game-service`、`gateway`。
- 目录：`exp4`
- 操作：
  1. 执行 `docker compose up -d`。
  2. 执行 `docker compose ps`。
  3. 执行 `docker compose logs -f`。
  4. 执行 `docker compose down`。
  5. 如需清理卷，执行 `docker compose down -v`。
- 成功标准：
  - 三个服务均为运行状态
  - 自动创建网络与数据卷
  - `down` 不删除卷，`down -v` 删除卷

### 实验五：多服编排与服务发现 (对应“编排”知识点)

- **测试目标：** 演示单体向微服务拆分后，演示如何通过编排工具一键拉起多个相互依赖的服务；在真实生产环境中，该机制可以扩展到成百上千个实例。

- **操作步骤：**
  1. 展示上一章写的“网关”、“战斗逻辑”、“存储”三个微服务。如果手动敲命令行，需要配置复杂的 IP 端口并启动三次。
  2. 展示一份 K8s 的 `Deployment` 和 `Service` YAML 声明文件。
  3. 执行一行指令：`kubectl apply -f game-cluster.yaml`。
  4. 通过网关发起战斗请求。
- **观察到什么算成功（直观感受）：**
  - **一键创世：** 执行指令后，控制台瞬间并排打印出 `Gateway-Pod`、`Game-Pod`、`Storage-Pod` 全部处于 `Running` 状态。
  - **自动发现：** 即使每次 Pod 启动的 IP 都不一样，网关依然能通过域名（如 `http://game-service`）成功把请求转发给战斗逻辑。学生能直观感受到什么是**“声明式编排”**。

### 实验六：K8s Service 内外部访问

- **对应页码**：第 45-47 页
- **测试目标**：演示 Service 既可以作为集群内部稳定入口，也可以通过 NodePort 暴露给集群外部客户端。

- **操作步骤：**
  1. 创建 ClusterIP Service：`kubectl apply -f game-service.yaml`
  2. 在集群内部启动 debug 容器：`kubectl run -it --rm debug --image=busybox -- /bin/sh`
  3. 在 debug 容器中访问：`wget http://game-service:8081/move`
  4. 创建 NodePort Service：`kubectl apply -f game-svc-external.yaml`
  5. 在集群外部访问：`curl http://<任意Node物理IP>:30080/move`
  
- **观察到什么算成功**：
  - 集群内部可以通过 `game-service:8081` 访问。
  - 集群外部可以通过 `NodeIP:30080` 访问。
  - 访问不同 Node 的物理 IP，效果一致，体现 Service 的转发能力。

### 实验七：弹性伸缩应对潮汐流量 (对应“弹性-伸缩调度”知识点)

- **测试目标：** 演示云计算最大的魅力：资源按需分配，应对突发的“公会战”或“节假日高峰”。

- **注意事项**：
  - metrics-server 已安装
  - Deployment 配置了 resources.requests.cpu
  - 压测接口能够实际消耗 CPU
  - HPA 扩缩容存在几十秒到数分钟延迟

- **操作步骤：**

  1. 在 K8s 中部署只有 1 个副本的 Game 服务，并配置 HPA（水平 Pod 自动扩缩容），设定 CPU 阈值为 50%。
  2. **制造流量海啸：** 教师运行一个压测脚本（模拟 1000 个玩家同时释放复杂技能），打满这 1 个服务的 CPU。
  3. 开启分屏，实时监控：`watch kubectl get pods`。
  4. 停止压测，等待几分钟。

- **观察到什么算成功（直观感受）：**

  - **自动扩容：** 压测开始后，学生会肉眼看到屏幕上的 Pod 数量从 1 个自动裂变成 2 个、4 个、8 个... 直到扛住所有流量。
  - **自动缩容：** 压测停止后，系统发现 CPU 闲置，又会自动把多余的 7 个副本销毁（节约云服务器成本）。完美展现了**“弹性计算”**的威力。

  *这里并没有明确提及VPA的配置，建议也配置一下演示实际的效果*

### 实验八：无状态网关与会话重连 (无感飘移)

- **对应页码**：第 63-67 页

- **测试目标**：验证将玩家会话（Session）外置到 Redis 后，网关实例的随机销毁与重启不会导致玩家数据丢失，实现“只会卡一下，不会掉进度”的无感切换。

- **操作步骤：**

  1. **部署环境**：在 K8s 中部署一个 Redis 实例和 2 个副本的网关（`Gateway-Pod`）。

  2. **建立连接**：客户端连接到 `Pod-1`，网关代码将玩家 ID 和对应的游戏服务器地址写入 Redis，并设置心跳。

  3. **模拟事故（缩容/崩溃）**：

     - 手动执行 `kubectl delete pod <Pod-1-Name>`，模拟网关突然崩溃。
     - 或者修改 `replicas` 从 2 变为 1，模拟缩容过程。

  4. **观察重连**：客户端检测到连接断开，自动发起重连，此时 K8s 将请求路由到存活的 `Pod-2`。

     *客户端可以是一个 Go 编写的命令行程序
     也可以是 WebSocket 客户端
     要求每 1 秒发送一次 heartbeat
     断线后携带 token 自动重连*

- **观察到什么算成功（直观感受）**：

  - **失败表现（有状态网关）**：如果网关把玩家数据存在内存 `map` 里，`Pod-1` 消失后，`Pod-2` 找不到玩家数据，玩家会强制退回到登录界面或发生回档。
  - **成功标准（无状态网关）**：`Pod-2` 启动后，根据客户端携带的 Token 从 Redis 中精准读出玩家状态并接管会话。玩家在客户端只会感受到大约 1-2 秒的闪断重连，随后游戏继续，进度完好无损。
  - **高级验证（优雅关闭）**：查看日志，确认 `preStop` 钩子生效（如 `sleep 30`），即 `Pod-1` 在被杀掉前给了正在进行的请求充足的处理时间。

### 实验九：Node 级扩缩与云资源协同 (基础设施弹性)

- **对应页码**：第 69-75 页 

  *该实验建议作为云环境选做实验。*

- **测试目标**：验证基础设施层（Infrastructure Layer）的自动扩缩机制 。通过 Cluster Autoscaler (CA) 演示在应用请求超出集群现有总容量时，如何自动调用云平台 API “买机器” 。

- **操作步骤：**

  1. **环境配置**：在集群中安装 Cluster Autoscaler 插件，并配置云厂商 API 权限（如 AccessKey） 。
  2. **设置安全边界**：划定节点组（Node Group），设定最小 1 台、最大 10 台的伸缩范围 。
  3. **触发扩容（资源挤爆）**：
     - 编写一个 `Deployment`，明确声明 `resources.requests` 为较大的资源值（如 `1Gi` 内存） 。
     - 猛增副本数（Replicas），直到集群现有 Node 无法承载更多的 Pod，导致 Pod 进入 `Pending` 状态 。
  4. **设置缩容保护**：
     - 为关键业务（如 Redis 缓存）配置 `PodDisruption Budget (PDB)`，要求最少 2 个实例存活 。
     - 为带数据的 Pod 加上 `safe-to-evict: "false"` 的注解 。

- **观察到什么算成功（直观感受）**：

  - **自动补位成功**：当 Pod 因为资源不足卡在 `Pending` 时，观察到云平台后台自动创建了新的虚拟机（Node）并加入集群，随后 `Pending` 的 Pod 自动在这些新机器上跑起来，实现“分钟级”扩容 。
  - **缩容保护成功**：当流量消退触发缩容时，CA 能够识别出带有 PDB 保护或 `safe-to-evict: false` 标记的节点并跳过它们 。你会在日志中看到“由于 Pod 保护限制，跳过该节点的回收”，证明关键业务不会因为追求省钱而意外中断 。

### 实验十：Serverless 事件驱动函数 (每日奖励结算)

- **对应页码**：第 79-83 页 
- **测试目标**：验证“按需实例化”与“用完即走”的特性，直观感受代码即服务（FaaS）、毫秒级计费以及零闲置成本的优势 。
- **操作步骤**：
  1. **编写函数代码**：参考 PPT 中的 Python 示例，实现一个模拟签到奖励的入口函数 `handler(event, context)` 。
  2. **声明式部署**：编写 `s.yaml` 配置文件，明确限定资源（如 `128MB` 内存）和触发器路径（如 `/signin`） 。使用工具执行 `s deploy` 一键上传云端 。
  3. **事件触发**：使用 `curl` 模拟玩家点击签到，向生成的 URL 发起 POST 请求 。
- **观察到什么算成功**：
  - **按需运行（Scale-to-Zero）**：通过云平台控制台观察，在没有 `curl` 请求时，系统没有任何运行实例，也不产生费用 。
  - **即时拉起（冷启动）**：观察请求响应时间。第一次调用时会有微小的加载延迟（冷启动），一旦环境预热，随后的调用将瞬间完成（热启动） 。
  - **自动扩容**：模拟极高并发的签到请求，观察平台如何瞬间自动产生上百个并发执行环境，无需人工预留机器 。

### 实验十一：探针检测与故障自愈 (对应“弹性-故障自愈”知识点)

**测试目标：** 演示即使程序员写了 Bug 导致进程假死，云原生架构也能让游戏“永不宕机”。

- **操作步骤：**
  1. 给 Game 服务配置存活探针（Liveness Probe，每 5 秒 ping 一次 `/health` 接口）。
  2. 正常运行时查看状态：`kubectl get pods`，显示 `STATUS: Running`, `RESTARTS: 0`。
  3. **制造车祸（模拟进程假死）：** 向 Game 服务发送一个特殊的“死锁”指令，让进程仍然存在，但 /healthz 接口不再响应或返回 500。
  4. 等待 15 秒，观察 K8s 的动作。
- **观察到什么算成功（直观感受）：**
  - 学生会看到，进程虽然假死了，但 15 秒后，K8s 的探针敏锐地发现了它没有心跳。
  - 终端上自动显示：该 Pod 被 K8s 强行杀掉并原地复活。再次执行 `kubectl get pods`，会看到 `RESTARTS` 变成了 `1`，状态重新回到 `Running`。
  - **升华总结：** 真正的生产自愈是带约束的（比如是不是只剩最后一个实例了？是不是有状态服务？）。
