## 一、实验目标

本实验目标是把"单地图、单节点、单机房"的战场系统，升级为"多地图并行 + 多服务节点协同"的分布式版本。

你需要完成两类能力：

**玩法递进**

- 地图不再是全可通行平面，而是包含障碍、走廊和不可通行区域的大地图
- 地图中可随机刷新 NPC 和宝物
- NPC 会自主移动、攻击玩家，死亡后掉落战利品
- 玩家可以攻击 NPC、拾取宝物、进入商店购买药剂和强化武器
- 世界首领在多张地图上均可见，并由全服玩家共同攻击

**架构递进**

- 支持多张地图并行运行
- 支持多服务节点协同处理
- 支持热数据和冷数据分层存储
- 支持节点故障、主从切换与状态恢复

实验重点不是"把功能堆出来"，而是明确以下四个分布式设计点：

1. 分布式扩展能力
2. 服务协同能力
3. 状态一致性保障
4. 故障容错与高可用

---

## 二、实验背景

Lab2 中，系统通常只包含一张对战地图，所有玩家和 NPC 的逻辑都在单个进程内完成。这种设计实现简单，但存在明显问题：

- 地图和玩家数量一多，单节点会成为瓶颈
- 一个节点故障，整张地图就无法继续服务
- 无法支持跨地图协同和跨节点共享对象
- 玩家退出重登后，状态恢复能力有限

因此在 Lab3 中，需要把系统拆成"网关 + 多地图节点 + 持久化存储"的结构，并且让它具备基本的分布式协同能力。

## 三、实验要求

### 1. 地图与玩法要求

1. 设计更大的地图，地图至少要有明显的障碍布局，不能所有格子都可通行
2. 地图中随机刷新 NPC，且 NPC 分布不能明显扎堆在单一区域
3. NPC 需要自行随机移动，并在合适条件下攻击玩家
4. 玩家可以攻击 NPC，NPC 死亡后应掉落战利品
5. 地图中随机刷新宝物，玩家可以拾取
6. 玩家按 `P` 可进入商店，使用战利品购买药剂或强化武器
7. 玩家死亡后需要经过倒计时自动复活，复活后可以继续移动和战斗
8. 世界首领需要在各地图上显示投影位置，并由所有地图上的玩家共享同一份血量

### 2. 分布式架构要求

1. 多张地图可以同时推进各自状态
2. 不同地图可以分配给不同节点处理
3. 网关负责统一接入、登录注册、指令路由和节点管理
4. 多节点之间要能共享全局对象，例如世界首领
5. 节点故障后，系统应尽量依靠快照和副本恢复服务

### 3. 数据持久化要求

系统需要区分热数据和冷数据：

**冷数据**

- 用户名、密码
- 历史战绩
- 上次退出时所在地图、坐标、生命、药剂、战利品等

**热数据**

- 当前在线会话
- 当前地图状态
- 地图检查点
- 世界首领当前状态

## 五、实验任务与评分

所有待完成的函数均位于 `student/cluster/cluster.go`，共 5 个核心 TODO。

### Lab3-1：`cluster.AttackBoss`　　**（3 分）**

完成世界首领的全服共享协同结算。

**要求**：

1. 不同地图、不同节点上的玩家，攻击的是同一只世界首领
2. 首领血量必须全局一致，使用集群锁保护单写者更新
3. 攻击结果要正确广播给所有在线会话，而不只是当前地图
4. 首领死亡时，记录终结玩家，结算全员奖励，并安排 15 秒后重生

### Lab3-2：`cluster.SwitchMap`　　**（3 分）**

完成跨地图切换和会话迁移。

**要求**：

1. 玩家切图时，从源节点安全移除玩家热状态
2. 根据 `owners` 路由将玩家挂到目标地图主节点
3. 更新 `session.MapID` / `session.NodeID`，后续指令路由正确
4. 倒地状态下不能切换地图，需返回提示并拒绝

### Lab3-3：`cluster.persistSessionState`　　**（3 分）**

完成热数据和冷数据落盘。

**要求**：

1. 冷数据：调用 `store.SaveProfile` 持久化账号信息、历史战绩、最近退出位置
2. 热数据：调用 `store.SaveHotSession` 保存当前地图、节点、坐标、生命值、会话版本
3. 玩家下次登录时，能够恢复上次退出时的地图和角色状态

### Lab3-4：`cluster.checkpointLoop`　　**（3 分）**

完成地图检查点的周期性复制。

**要求**：

1. 使用 `time.NewTicker` 每 2 秒触发一次
2. 从 `owners` 找到每张地图当前主节点，抓取地图快照
3. 同时调用 `store.SaveCheckpoint` 写入磁盘，并通过 `replica.StoreReplica` 同步到副本节点内存
4. 跳过已故障（`!IsHealthy()`）的节点，避免把坏状态扩散

### Lab3-5：`cluster.handleNodeFailure`　　**（3 分）**

完成故障切换和主从提升。

**要求**：

1. 找到故障节点承载的所有主地图
2. 选择对应副本节点，调用 `replica.Promote` 提升为新主节点
3. 更新 `owners` / `replicas` 元数据
4. 修正所有受影响玩家会话的 `NodeID`，并广播故障切换事件

## 七、分布式设计建议

这一部分不是唯一答案，但你实现时至少要满足下面的设计原则。

### 1. 分布式扩展能力

建议把地图按 `mapID` 分片，不同地图由不同节点负责。这样可以把地图内的刷新、移动、战斗循环拆开并行运行，避免所有逻辑都压在一个世界对象上。

### 2. 服务协同能力

客户端不要直接连具体地图节点，而是统一连接网关。网关负责：登录注册、用户会话管理、指令转发、地图切换、节点健康检查、故障切换后的重新路由。

### 3. 状态一致性保障

至少要保证同一张地图在同一时刻只有一个主节点可以写。否则会出现：同一玩家状态被重复修改、同一 NPC 被不同节点重复结算、世界首领血量不一致。

建议使用"单写者 + 周期复制"的思路：地图内状态由主节点独占写入，副本只接收快照，故障后再由副本提升为主。

### 4. 故障容错与高可用

建议通过心跳或管理命令模拟节点故障。一旦主节点失效，应尽量做到：读取最近检查点恢复地图，在副本节点上重新接管，玩家无须重新注册。

## 八、运行方式

> 平台兼容说明：
> Lab3 的服务端、客户端和管理命令可在 Windows、macOS、Linux 下运行。
> 客户端的即时按键输入已经改为跨平台实现，不再依赖 `stty` 和 `/dev/tty`。

### 1. 启动服务端

```bash
cd Lab3/student
go run ./cmd/server
```

默认网关地址为 `127.0.0.1:9310`。

### 2. 启动客户端

```bash
cd Lab3/student
go run ./cmd/client
```

登录后常用按键如下：

| 键 | 行动 |
|----|------|
| `W A S D` | 移动 |
| `J` | 普通攻击 |
| `K` | 使用药剂 |
| `B` | 攻击世界首领 |
| `P` | 打开或关闭商店 |
| `1 2 3` | 切换地图 |
| `Q` | 退出游戏 |

### 3. 管理命令

```bash
cd Lab3/student
go run ./cmd/admin 状态
go run ./cmd/admin 故障 node-a
go run ./cmd/admin 恢复 node-a
```

也可以显式指定网关地址：

```bash
go run ./cmd/admin 故障 node-a 127.0.0.1:9310
```
