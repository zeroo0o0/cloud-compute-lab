# Lab3：分布式战场与多服协同实验指导

## 一、实验目标

本实验在 Lab2 的基础上继续递进，目标是把“单地图、单节点、单机房”的战场系统，升级为“多地图并行 + 多服务节点协同”的分布式版本。

你需要完成两类能力：

1. 玩法递进
   - 地图不再是全可通行平面，而是包含障碍、走廊和不可通行区域的大地图
   - 地图中可随机刷新 NPC 和宝物
   - NPC 会自主移动、攻击玩家，死亡后掉落战利品
   - 玩家可以攻击 NPC、拾取宝物、进入商店购买药剂和强化武器
   - 世界首领在多张地图上均可见，并由全服玩家共同攻击

2. 架构递进
   - 支持多张地图并行运行
   - 支持多服务节点协同处理
   - 支持热数据和冷数据分层存储
   - 支持节点故障、主从切换与状态恢复

实验重点不是“把功能堆出来”，而是明确以下四个分布式设计点：

1. 分布式扩展能力
2. 服务协同能力
3. 状态一致性保障
4. 故障容错与高可用

---

## 二、实验背景

Lab2 中，系统通常只包含一张对战地图，所有玩家和 NPC 的逻辑都在单个进程内完成。这种设计实现简单，但存在明显问题：

1. 地图和玩家数量一多，单节点会成为瓶颈
2. 一个节点故障，整张地图就无法继续服务
3. 无法支持跨地图协同和跨节点共享对象
4. 玩家退出重登后，状态恢复能力有限

因此在 Lab3 中，需要把系统拆成“网关 + 多地图节点 + 持久化存储”的结构，并且让它具备基本的分布式协同能力。

## 三、实验要求

### 1. 地图与玩法要求

你需要在 Lab2 的基础上，支持以下玩法：

1. 设计更大的地图，地图至少要有明显的障碍布局，不能所有格子都可通行
2. 地图中随机刷新 NPC，且 NPC 分布不能明显扎堆在单一区域
3. NPC 需要自行随机移动，并在合适条件下攻击玩家
4. 玩家可以攻击 NPC，NPC 死亡后应掉落战利品
5. 地图中随机刷新宝物，玩家可以拾取
6. 玩家按 `P` 可进入商店，使用战利品购买药剂或强化武器
7. 玩家死亡后需要经过倒计时自动复活，复活后可以继续移动和战斗
8. 世界首领需要在各地图上显示投影位置，并由所有地图上的玩家共享同一份血量

### 2. 分布式架构要求

系统需要从“单地图、单节点处理”升级为“多地图并行 + 多节点协同”，并满足以下目标：

1. 多张地图可以同时推进各自状态
2. 不同地图可以分配给不同节点处理
3. 网关负责统一接入、登录注册、指令路由和节点管理
4. 多节点之间要能共享全局对象，例如世界首领
5. 节点故障后，系统应尽量依靠快照和副本恢复服务

### 3. 数据持久化要求

系统需要区分热数据和冷数据：

1. 冷数据
   - 用户名、密码
   - 历史战绩
   - 上次退出时所在地图、坐标、生命、药剂、战利品等

2. 热数据
   - 当前在线会话
   - 当前地图状态
   - 地图检查点
   - 世界首领当前状态

## 五、你需要完成的核心 TODO

本实验已经保留了大部分骨架代码，但在 `student/cluster/cluster.go` 中抽出了 5 个最关键的分布式 TODO。测试失败时，也会直接给出对应标签。

### Lab3-1：`cluster.AttackBoss`

你需要完成世界首领的全服共享协同结算。

要求：

1. 不同地图、不同节点上的玩家，攻击的是同一只世界首领
2. 首领血量必须全局一致
3. 攻击结果要正确广播给所有相关玩家
4. 首领死亡时，要记录终结玩家，并生成后续恢复或重生逻辑

### Lab3-2：`cluster.SwitchMap`

你需要完成跨地图切换和会话迁移。

要求：

1. 玩家切图时，从原地图安全移除
2. 玩家状态正确迁移到目标地图
3. 网关后续指令能够路由到正确地图
4. 倒地状态下不能切换地图

### Lab3-3：`cluster.persistSessionState`

你需要完成热数据和冷数据落盘。

要求：

1. 在线状态需要写入热数据
2. 玩家退出或关键状态变化时，需要同步冷数据
3. 玩家下次登录时，能够恢复上次退出时的地图和角色状态

### Lab3-4：`cluster.checkpointLoop`

你需要完成地图检查点复制。

要求：

1. 主节点周期性生成地图快照
2. 快照要同步到副本节点或热数据目录
3. 快照应覆盖地图中的关键状态，例如玩家、NPC、宝物、首领相关状态等

### Lab3-5：`cluster.handleNodeFailure`

你需要完成故障切换和主从提升。

要求：

1. 节点故障后，系统能够识别失效节点
2. 使用最近检查点恢复地图
3. 将副本节点提升为新的主节点
4. 将玩家会话路由切换到新的主节点

## 六、分布式设计建议

这一部分不是唯一答案，但你实现时至少要满足下面的设计原则。

### 1. 分布式扩展能力

建议把地图按 `mapID` 分片，不同地图由不同节点负责。这样可以把地图内的刷新、移动、战斗循环拆开并行运行，避免所有逻辑都压在一个世界对象上。

### 2. 服务协同能力

客户端不要直接连具体地图节点，而是统一连接网关。网关负责：

1. 登录注册
2. 用户会话管理
3. 指令转发
4. 地图切换
5. 节点健康检查
6. 故障切换后的重新路由

### 3. 状态一致性保障

至少要保证同一张地图在同一时刻只有一个主节点可以写。否则会出现：

1. 同一玩家状态被重复修改
2. 同一 NPC 被不同节点重复结算
3. 世界首领血量不一致

建议使用“单写者 + 周期复制”的思路：

1. 地图内状态由主节点独占写入
2. 副本只接收快照或复制数据
3. 故障后再由副本提升为主

### 4. 故障容错与高可用

建议通过心跳或管理命令模拟节点故障。一旦主节点失效，应尽量做到：

1. 读取最近检查点恢复地图
2. 在副本节点上重新接管地图
3. 玩家无须重新注册，尽量只需继续操作

## 七、运行方式

### 1. 启动服务端

在 `student/` 目录下运行：

```bash
cd Lab3/student
go run ./cmd/server
```

默认网关地址为 `127.0.0.1:9310`。

### 2. 启动客户端

```bash
cd Lab3/student
go run ./cmd/client
```

客户端支持两种入口：

1. 单机测试
   - 默认连接 `127.0.0.1:9310`

2. 连接指定网关
   - 手动输入网关 IP 和端口

登录后常用按键如下：

1. `W A S D`
   - 移动
2. `J`
   - 普通攻击
3. `K`
   - 使用药剂
4. `B`
   - 攻击世界首领
5. `P`
   - 打开或关闭商店
6. `1 2 3`
   - 切换地图
7. `Q`
   - 退出游戏

### 3. 管理命令

可用管理命令如下：

```bash
cd Lab3/student
go run ./cmd/admin 状态
go run ./cmd/admin 故障 node-a
go run ./cmd/admin 恢复 node-a
```

也可以显式指定网关地址：

```bash
go run ./cmd/admin 状态 127.0.0.1:9310
go run ./cmd/admin 故障 node-a 127.0.0.1:9310
go run ./cmd/admin 恢复 node-a 127.0.0.1:9310
```

## 九、调试建议

### 1. 端口占用

测试脚本默认使用以下端口：

1. `9310`
   - 网关
2. `9311`
   - `node-a`
3. `9312`
   - `node-b`
4. `9313`
   - `node-c`

如果这些端口已被旧进程占用，`./run_test.sh` 会直接报错并提示你先关闭残留服务。

### 2. 学生版测试失败时的日志

如果 `student/` 服务启动失败，或者测试执行失败，脚本会自动把服务端日志打印出来，便于你判断问题发生在：

1. 启动阶段
2. 登录注册阶段
3. 切图阶段
4. 世界首领协同阶段
5. 故障切换阶段

### 3. 数据目录

默认情况下，程序会在 `student/data/` 下使用以下目录：

1. `data/cold/`
   - 冷数据
2. `data/hot/`
   - 热数据
3. `data/hot/checkpoints/`
   - 地图检查点

测试脚本会使用临时数据目录，不会污染你的日常实验数据。

如果你想临时指定数据目录，也可以在启动服务端时设置：

```bash
LAB3_DATA_ROOT=/tmp/lab3_demo go run ./cmd/server
```
